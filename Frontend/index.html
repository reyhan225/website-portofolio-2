CVE-2023-XXXX: SQL Injection via Unsanitized Search Parameter in University Portal

During a penetration test of an internal university portal application, I identified a critical SQL injection vulnerability in the student search endpoint. The application passed user-supplied input directly into a dynamic SQL query without any parameterization or sanitization — exposing the entire student database to extraction.

🔓 Vulnerability Analysis

Type: SQL Injection (Error-Based & UNION-Based)

Vector: The /search?q= parameter was concatenated directly into a raw SQL query. No input validation, no prepared statements, no WAF.

Vulnerable code pattern:

// VULNERABLE (never do this) $q = $_GET['q']; $sql = "SELECT * FROM students WHERE name = '$q'"; // Attacker input: ' OR '1'='1 // Result: full table dump

Impact: Attacker could extract all student names, NIM numbers, emails, grades, and admin credentials using UNION-based injection. Boolean-blind techniques also confirmed feasibility without error output.

✅ Mitigation Strategy

1. Parameterized Queries (Primary Fix): Replace all dynamic SQL with PDO prepared statements or an ORM layer.

// SECURE - Prepared Statement $stmt = $pdo->prepare( "SELECT * FROM students WHERE name = ?" ); $stmt->execute([$_GET['q']]);

2. Input Validation: Whitelist allowed characters for search fields (alphanumeric + space only).

3. Least Privilege: Database user account should only have SELECT permission on necessary tables — not INSERT/DROP/UPDATE.

4. Error Handling: Suppress database error messages from end users. Log internally only.

5. WAF Layer: Deploy ModSecurity with OWASP Core Rule Set as a defense-in-depth measure.
